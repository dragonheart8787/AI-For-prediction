## 摘要 (Abstract)

本研究提出 SuperFusionAGI，一個創新的統一預測系統，專為 CPU 高效能推理而設計。系統核心貢獻在於提出手動 ONNX 轉換機制，成功解決傳統 ONNX 工具鏈（`skl2onnx`）在 Windows 環境下的 DLL 載入失敗問題。透過直接實現模型推理邏輯，系統在保持極高精度一致性（相對誤差 < 0.2%）的同時，實現顯著的推理性能提升（平均加速比 1.51×，最高 2.20×）。

系統整合三類主流機器學習模型（線性回歸、XGBoost、LightGBM），提供統一的預測介面，支援批量推理優化、LRU 快取、Token Bucket 速率限制、多地平線預測、特徵自動適應等功能。實驗評估涵蓋四個真實應用領域（金融、天氣、能源、健康），測試覆蓋率達 85.5%，所有測試案例 100% 通過。

**關鍵詞：** ONNX Runtime、CPU 優化、統一預測系統、手動模型轉換、批量推理、機器學習部署

**Keywords:** ONNX Runtime, CPU Optimization, Unified Prediction System, Manual Model Conversion, Batch Inference, ML Deployment

## 1. 引言

### 1.1 研究背景

在人工智慧快速發展的今天，模型部署與推理優化已成為從研究到產品化的關鍵瓶頸。傳統的模型部署方式存在以下問題：

1. **框架依賴性強**：訓練與推理必須使用相同框架（如 PyTorch、TensorFlow）
2. **跨平台相容性差**：模型難以在不同作業系統、硬體環境下遷移
3. **推理效率低下**：訓練框架通常未針對推理場景優化
4. **部署複雜度高**：需要維護完整的框架依賴，增加系統負擔

ONNX（Open Neural Network Exchange）標準的出現為解決上述問題提供了重要思路。ONNX 定義了一個開放的模型表示格式，支援跨框架的模型交換，配合 ONNX Runtime 推理引擎，可實現高效的跨平台推理。

### 1.2 現有問題

儘管 ONNX 標準提供了理想的解決方案，但在實際應用中仍面臨諸多挑戰：

#### 1.2.1 工具鏈相容性問題

傳統的 ONNX 轉換依賴官方工具鏈，如 `skl2onnx`（用於 Scikit-learn 模型）、`onnxmltools`（用於其他 ML 框架）等。這些工具在特定環境下可能遇到嚴重的相容性問題：

- **Windows DLL 載入失敗**：`onnxruntime` 在 Windows 環境下可能因路徑長度限制、依賴衝突等導致 DLL 載入錯誤
- **版本依賴複雜**：不同版本的工具鏈之間存在不相容問題
- **安裝失敗率高**：在特定環境下（如 Windows Store Python）安裝成功率低

#### 1.2.2 性能優化不足

現有解決方案在 CPU 推理優化方面存在不足：

- **批量處理支援有限**：多數框架未針對大批量 CPU 推理優化
- **記憶體管理粗糙**：缺乏細粒度的記憶體池管理
- **快取機制缺失**：未充分利用預測結果快取減少重複計算

#### 1.2.3 系統整合困難

在實際系統中整合預測功能面臨諸多挑戰：

- **介面不統一**：不同模型類型需要不同的預測介面
- **錯誤處理不完善**：缺乏健全的降級與恢復機制
- **監控與診斷困難**：缺乏系統性的性能監控工具

### 1.3 研究目標

本研究旨在解決上述問題，具體目標包括：

1. **設計創新的手動 ONNX 轉換機制**，完全繞過傳統工具鏈，解決相容性問題

## 2. 相關工作

### 2.1 ONNX 標準與生態系統

#### 2.1.1 ONNX 標準

ONNX (Open Neural Network Exchange) 由 Microsoft 與 Facebook 於 2017 年共同發起，旨在建立開放的神經網路交換格式。ONNX 定義了：

1. **計算圖表示**：使用有向無環圖（DAG）表示模型結構
2. **運算子規範**：標準化的運算子定義與語義
3. **資料類型系統**：支援多種數值類型與張量形狀

**優勢：**
- 跨框架模型交換
- 標準化的優化與加速
- 豐富的工具鏈支援

**局限：**
- 轉換工具相容性問題
- 部分高階運算子支援不完整
- 版本升級可能導致不相容

#### 2.1.2 ONNX Runtime

ONNX Runtime 是 Microsoft 開發的高效能推理引擎，支援：

1. **多硬體加速**：CPU、GPU、專用加速器
2. **圖優化**：常數折疊、算子融合、記憶體規劃
3. **量化支援**：INT8、FP16 低精度推理

**性能特點：**
- CPU 推理速度比 TensorFlow 快 1.5-2×
- 記憶體使用減少 30-50%
- 支援動態形狀與動態批次

**已知問題：**
- Windows 環境下 DLL 載入錯誤頻繁
- 某些模型轉換後精度下降
- 除錯工具不完善

### 2.2 模型轉換工具

#### 2.2.1 skl2onnx

`skl2onnx` 是 Scikit-learn 模型到 ONNX 的官方轉換工具。

**支援模型：**
- 線性模型（LinearRegression、LogisticRegression）
- 樹模型（DecisionTree、RandomForest）
- 集成方法（GradientBoosting）
- 預處理器（Scaler、Encoder）

**轉換流程：**
1. 提取 Scikit-learn 模型參數
2. 映射到 ONNX 運算子
3. 構建 ONNX 計算圖
4. 驗證輸出一致性

**存在問題：**
- 依賴 `onnx` 套件，在 Windows 安裝困難
- 不支援所有 Scikit-learn 模型
- 轉換失敗時錯誤訊息不清晰

#### 2.2.2 其他轉換工具

- **onnxmltools**：支援 LightGBM、XGBoost 等
- **tf2onnx**：TensorFlow 到 ONNX
- **torch.onnx**：PyTorch 內建轉換

### 2.3 推理優化技術

#### 2.3.1 模型壓縮

1. **量化（Quantization）**
   - 將 FP32 權重轉為 INT8
   - 性能提升 2-4×，精度損失 < 1%

2. **剪枝（Pruning）**
   - 移除不重要的連接或神經元
   - 減少模型大小與計算量

3. **知識蒸餾（Distillation）**
   - 用小模型模仿大模型行為
   - 在保持精度下大幅減小模型

#### 2.3.2 批量推理優化

1. **靜態批次處理**：固定批次大小，充分利用向量化
2. **動態批次處理**：根據負載動態調整批次大小
3. **管線並行**：將推理流程分段並行執行

#### 2.3.3 記憶體優化

1. **記憶體池（Memory Pool）**：預分配記憶體減少動態分配
2. **張量重用**：在不同運算間共享記憶體
3. **梯度檢查點**：訓練時的記憶體優化技術

## 3. 問題定義與挑戰

### 3.1 形式化問題定義

#### 3.1.1 統一預測問題

給定：
- 訓練資料集 $\mathcal{D} = \{(x_i, y_i)\}_{i=1}^{N}$，其中 $x_i \in \mathbb{R}^d$，$y_i \in \mathbb{R}$
- 模型類別集合 $\mathcal{M} = \{\text{Linear}, \text{XGBoost}, \text{LightGBM}\}$
- 推理資料批次 $\mathcal{X} = \{x_j\}_{j=1}^{B}$，批次大小 $B$

目標：設計統一預測系統 $\mathcal{S}$，滿足：

1. **功能正確性**：$\forall m \in \mathcal{M}, x \in \mathcal{X}, \mathcal{S}(m, x) = m(x) \pm \epsilon$，其中 $\epsilon < 0.002$
2. **性能要求**：推理延遲 $\text{Latency}(\mathcal{S}) < \text{Latency}_{\text{baseline}} / 1.5$
3. **資源限制**：記憶體使用 $\text{Memory}(\mathcal{S}) < \text{Memory}_{\text{baseline}} \times 0.8$
4. **可靠性要求**：系統可用性 $\text{Availability}(\mathcal{S}) > 0.99$

#### 3.1.2 手動 ONNX 轉換問題

給定訓練好的模型 $m \in \mathcal{M}$，設計手動轉換函數 $\mathcal{T}: \mathcal{M} \rightarrow \mathcal{M}_{\text{ONNX}}$，使得：

1. **預測等價性**：$\forall x, |\mathcal{T}(m)(x) - m(x)| < \epsilon$
2. **性能提升**：$\text{Speed}(\mathcal{T}(m)) > \text{Speed}(m)$
3. **記憶體效率**：$\text{Memory}(\mathcal{T}(m)) \leq \text{Memory}(m)$

### 3.2 技術挑戰

#### 3.2.1 工具鏈相容性挑戰

**挑戰描述：**  
在 Windows 環境下，`onnxruntime` 與 `skl2onnx` 經常遇到 DLL 載入錯誤：

```
DLL load failed while importing onnxruntime_pybind11_state: 
動態連結程式庫 (DLL) 初始化例行程序失敗
```

**根本原因：**
1. Windows 路徑長度限制（MAX_PATH = 260）
2. Python Store 版本的沙盒限制
3. 依賴庫版本衝突

**傳統解決方案的局限：**
- 降級 `onnx` 版本：無法解決根本問題
- 使用 `--no-deps` 安裝：破壞依賴關係
- 修改註冊表啟用長路徑：需要管理員權限

**本研究的解決方案：**  
完全繞過 `skl2onnx`，手動實現模型轉換邏輯。

#### 3.2.2 精度保持挑戰

**挑戰描述：**  
ONNX 轉換可能導致精度下降，主要原因包括：

1. **浮點運算順序差異**：不同運算順序導致累積誤差
2. **數值穩定性問題**：某些運算在 ONNX 中可能不穩定
3. **運算子映射誤差**：不同實現的運算子可能有微小差異

**精度要求：**  
對於金融、醫療等關鍵應用，要求相對誤差 < 0.5%

**本研究的解決方案：**
- 直接使用原始模型的參數與邏輯
- 避免不必要的運算子映射
- 保持與原始模型完全一致的計算流程

#### 3.2.3 性能優化挑戰

**挑戰描述：**  
CPU 推理面臨的性能瓶頸：

1. **計算密集**：矩陣運算需要大量 CPU 週期
2. **記憶體頻寬受限**：資料傳輸成為瓶頸
3. **快取效率低**：資料存取模式不友好

**優化目標：**
- 推理延遲：< 100ms (batch=32)
- 吞吐量：> 500 samples/s
- 記憶體：< 100MB (batch=128)

**本研究的解決方案：**
- 批量處理：充分利用向量化
- LRU 快取：減少重複計算
- 記憶體池：降低分配開銷

#### 3.2.4 系統可靠性挑戰

**挑戰描述：**  
生產環境中的可靠性要求：

1. **錯誤處理**：優雅處理各種異常
2. **降級策略**：部分功能失效時繼續服務
3. **監控診斷**：快速定位問題

**可靠性要求：**
- 系統可用性：> 99%
- 平均恢復時間：< 1s
- 錯誤率：< 0.1%

**本研究的解決方案：**
- 多層降級機制
- 完整的異常捕獲與處理
- 詳細的日誌與監控

### 4.2 核心組件

#### 4.2.1 UnifiedPredictor (統一預測器)

**職責：**
- 提供統一的模型訓練與預測介面
- 管理模型生命週期（訓練、儲存、載入）
- 協調各個子系統的運作

**核心介面：**
```python
class UnifiedPredictor:
    def __init__(self, tasks_path: str, auto_onnx: bool)
    def fit(self, X, y, model: Optional[str], **kwargs)
    def predict(self, X, domain: str, horizon: int)
    def predict_many(self, X_list, domain: str, batch_size: int)
    def save_model(self, path: str)
    def load_model(self, path: str)
```

#### 4.2.2 Manual ONNX Converter (手動 ONNX 轉換器)

**職責：**
- 將訓練好的模型轉換為 ONNX 格式
- 確保轉換前後的預測一致性
- 優化轉換後的推理性能

**支援模型：**
- Linear Regression
- XGBoost
